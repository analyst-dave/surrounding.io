## 🖼️ Part 1: The CV Architecture (The "Point Generator")

To automatically generate the maximum number of tags, the system must run multiple, redundant CV models simultaneously on the uploaded image.

| System Module | Function | Example Output/Tag Type | Point Weight Impact |
| :--- | :--- | :--- | :--- |
| **1. Object Detection** | Finds and identifies objects within the image boundaries. | `#dog`, `#bicycle`, `#building`, `#book` | Low (Base weight) |
| **2. Semantic Segmentation** | Breaks the image down into labeled physical regions (e.g., sky, grass, skin). | `#sky`, `#foliage`, `#portrait`, `#shadow` | Medium (High detail, medium weight) |
| **3. Image Classification** | Determines the overall mood, style, or subject genre. | `#sunset`, `#urban`, `#macro`, `#moody`, `#vibrant` | Medium (Thematic weight) |
| **4. Captioning (NLP/CV)** | Generates a natural language description, which is then tokenized into tags. | "A runner during the golden hour." $\rightarrow$ `#running`, `#runner`, `#goldenhour`, `#action` | High (Thematic/Narrative weight) |
| **5. Metadata/AI Guessing** | Analyzes technical aspects (lens flare, depth of field, specific colors). | `#bokeh`, `#lowlight`, `#analog`, `#anamorphic` | Very High (Technical/Niche weight) |

### 🎯 The System's "Hack" (What makes the user succeed)

The user needs to understand that the system is most activated by **complexity, variety, and depth.**

*   **Best Photos:** Busy street scenes, complex group portraits, wide-angle landscapes with multiple foreground/background elements, images with strong contrast, mixed light sources, or visible depth of field.
*   **Worst Photos:** Flat lighting, solid colors, isolated single subjects (e.g., a white wall, a simple apple on a plate).

## 🛠️ Backend Logic: The Scoring Engine

The scoring engine must iterate over every detected tag and apply a cumulative, weighted formula to determine the final points.

### 1. Tag Detection Pipeline (Microservice)

This service accepts an image and must return a list of raw tags, categorized by confidence.

### 2. Point Calculation Formula (Core Logic)

For each `tag`, the system calculates a raw point value:

$$
\text{Points}_{\text{tag}} = \text{Weight}_{\text{tag}} \times \text{Confidence}_{\text{tag}} \times \text{Bonus}_{\text{tag}}
$$

Where:

*   **$\text{Weight}_{\text{tag}}$ (Difficulty/Utility Weight):** A fixed, database-driven weight based on the tag's inherent value.
    *   *Example:* `#portrait` = 5.0 (High Utility). `#blue` = 1.0 (Common).
*   **$\text{Confidence}_{\text{tag}}$ (Model Confidence):** The confidence score returned by the image recognition model (0.0 to 1.0).
*   **$\text{Bonus}_{\text{tag}}$ (Contextual Bonus):** A variable multiplier applied based on how *unique* the tag is in the user's history or how well it combines with other tags.
    *   $\text{Bonus}_{\text{tag}} = 1.0 + (0.1 \times \text{Uniqueness\_Score})$

### 3. Total Score Calculation

The final point total is the sum of all calculated tag points.

$$
\text{Total Points} = \sum_{i=1}^{N} \text{Points}_{\text{tag}_i}
$$

## 📚 Data Management & Scalability

### Database Schema Needs:

1.  **`Tags` Table:** `tag_name`, `base_weight`, `type` (e.g., `mood`, `subject`, `color`).
2.  **`Users` Table:** `user_id`, `historical_tag_profile` (JSON/Vector DB for rapid similarity lookup).
3.  **`Analysis_History` Table:** `analysis_id`, `user_id`, `image_url`, `score_data` (JSON blob of the result).

### Scalability Considerations:

*   **Asynchronous Processing:** Image analysis must never block the main API thread. Use a message queue (e.g., Redis Queue, Kafka) to offload the computationally intensive image processing to dedicated workers.
*   **Vector Database:** For calculating the `Uniqueness_Score`, the system must efficiently query the User's historical tag vector against the current tag vector to gauge novelty.

***This structured approach ensures that every component—from the raw mathematical model to the polished user experience—is designed to deliver a cohesive and engaging product.***
